--- abstract

This document defines a new KeyPurposeId for attestation purposes.
The inclusion of this KeyPurposeId in an X.509 ({{!RFC5280}}) certificate
ExtendedKeyUsage extension indicates that the certificate public key
may be used for validation of attestation information.

# Introduction

Key purpose identifiers (KeyPurposeId) may be added to the Extended Key
Usage (EKU) extension of X.509 certificates to express approved
usage(s) of the certified key.  This specification introduces the
KeyPurposeId `id-kp-attestationKey` to the enable the marking of a
certificate containing the public key portion of an Attestation Key
(AK) key pair.

The use of `id-kp-attestationKey` is meant to be attestation-system
agnostic and means that the key may be used with some attestation
technology.  Some certificates may include other KeyPurposeId's that
are tied to one specific attestation technogy or infrastructure.

## Attestation

"Attestation",  as generally understood,  is one entity making a statement
of the truth or correctness of a given set of data.  In the instant
case, attestation refers to a process executed with a security module,
over data held by or known by the module, but that may be
cryptographically verified by other entities.

See {{RATS}} and {{TPM}} for some examples of worked attestation
systems.

"Security Module" as used here means some form of secure processing
element such as a Hardware Security Module or a hybrid
(hardware/firmware) system like the Trusted Execution Environment,
usually with secure storage of some sort.  Various relying parties
will have their own requirements for what constitutes a "trustworthy"
security module.  Specification of those is left to the policies of
the relying parties.

## Attestation Keys

An AK private key is used either to sign attestation data, or to
execute other methods where the AK public key may then be used to
verify or validate the attestation data or process.  Digital signature
sign/verify processing will be the most common method of validating
attestation data, but "proof by decryption" or other techniques may be
acceptable or necessary in some instances.

AKs may be either be generated by an end-user or by the manufacturer
of a given security module.  Manufacturers SHOULD provide
manufacturer-signed certificates for manufacturer generated AKs.
Certificates containing end-user generated AK public keys MAY be
provided by any CA acceptable to the relying parties for a given
attestation.

# Issuance

## Attestation Key Attributes

An issuing CA MUST NOT issue a certificate containing an
`id-kp-attestationKey` KeyPurposeId unless the CA can verify the
public key/AK meets the following minimum requirements:

* The AK was generated within a trusted security module, or was
  generated and loaded to the security module during a secure
  manufacturing process.  Derivation from a per-module seed meets this
  requirement.
* The AK is bound the security module, and may not be extracted,
  backed up or cloned to another security module.
* The AK is a `restricted` key in that it cannot be used for any
  purpose other than attestation. This restriction is enforced by the
  security module which will usually pre-format the data to be attested.
* (Optional) The certificate requester has the rights to use the AK for
  attestation. Note that the  rights-to-use for manufacturer-installed
  AKs may encompass all of the end users of the security module, while
  user-generated AKs may be restricted to a particular user or a
  group.

## KeyUsage Bits

The `KeyUsage` extension (KU) complements the EKU and must have a
single bit set that is appropriate for the attestation technology
represented by the key.

Most attestation technologies will be based on a sign/verify
model. Keys meant for use with sign/verify MUST only have the
`digitalSignature` bit set.

Other attestation schemes (e.g. TPM's TPM2_ActivateCredential) use a
"proof by decryption" model.  Keys meant for use with this model MUST
only have the `encipherOnly` bit or the `decipherOnly` bit set.

## Other KeyPurposeId

An AK certificate's EKU SHOULD have a single KeyPurposeId of
`id-kp-attestationKey`.  As noted above, the exception to this
requirement is for the use of other proprietary/non-proprietary
KeyPurposeId's that indicate more specificity to the AK type.

## ExtendedKeyUsage Criticality

An EKU that includes only a single KeyPurposeId of
`id-kp-attestationKey` SHOULD be marked as `critical`.  The reasoning
here is that the usage of a true attestation key is constrained by the
security module it's contained by.  Presenting non-attestation data
for validation by this AK public key should ALWAYS fail.

An EKU that includes a KeyPurposeId of `id-kp-attestationKey` along
with other attestation-related KeyPurposeId SHOULD NOT be marked as
critical. 

# Reliance

A relying party SHOULD NOT trace a chain of trust for the validation
of attestation data through an end-entity certificate that does not
contain the `id-kp-attestationKey` KeyPurposeId.

The attestation scheme MUST be consistent with the KeyUsage bit
settings or the validation fails.

A relying party MUST NOT accept an AK certificate unless it has previously
configured the certification path for the AK to be valid for
attestations. I.e., relying parties should not accept AK certificates
unless they know that the CA's along the path have enforced the
issuance criteria (above) for an AK certificate.

# Security Considerations

{{!RFC5280}} introduces security considerations that are applicable to this specification.
The EKU purpose `id-kp-attestationKey` does not introduce additional security risks.
On the other hand, the adoption of this extended key purpose mitigates specific cross-protocol attacks in
relation to use of Attestation Keys.

# Appendix A. ASN.1 Module
{:numbered="false"}

The following module adheres to ASN.1 specifications [X.680] and [X.690].
It defines the OID used for Attestation Key Extended Key Usage.

~~~asn1
  AttestationEKU-2025 { iso(1) identified-organization(3) dod(6) internet(1)
       security(5) mechanisms(5) pkix(7) id-mod(0)
       id-mod-attestation-eku-2025(TBD0) }

  DEFINITIONS EXPLICIT TAGS ::=

  BEGIN

  -- EXPORTS ALL --

  IMPORTS
  id-kp
  FROM PKIX1Explicit-2009
         {iso(1) identified-organization(3) dod(6) internet(1)
          security(5) mechanisms(5) pkix(7) id-mod(0)
          id-mod-pkix1-explicit-02(51)}

  KeyPurposeId
  FROM PKIX1Implicit-2009
       {iso(1) identified-organization(3) dod(6) internet(1) security(5)
         mechanisms(5) pkix(7) id-mod(0) id-mod-pkix1-implicit-02(59)}
;
   


  -- Attestation Key Extended Key Usage --

  id-kp-attestationKey KeyPurposeId ::= { id-kp XX }

  END
~~~
